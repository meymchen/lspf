# Migrating from lspf 0.1 to 0.2

This guide ports an lspf 0.1.x server to the 0.2 model locked by ADRs
0017â€“0020. It is self-contained: you do not need to read the ADRs to
migrate.

## What 0.2 removes

lspf 0.2 removes the `LanguageServer` trait and its `documents()` getter
outright. There is no long-lived adapter and no deprecation cycle: the old
trait is gone in the 0.2 release, not phased out.

Rationale for removing without an adapter:

- lspf is still 0.x, where breaking changes are sanctioned;
- an adapter would force the framework to run two lifecycle and
  document-ownership models side by side; and
- the current example and downstream surface is small enough to migrate
  directly.

In place of the trait, 0.2 registers typed handlers on a builder:

- `Server::builder(state)` wraps your state in one `Arc<S>` shared by every
  handler for the connection;
- `feature` / `request` / `notification` / `command` register handlers with
  native Rust signatures; and
- `build()` validates the registrations and returns `Result<Server<S>,
  BuildError>`.

The examples below are the call shapes locked by ADRs 0017â€“0020; a few
details that the ADRs deliberately defer (the concurrency-limit tuning knob,
the read-only `DocumentsView`, the retained `serve(server, transport)`
helper) follow the 0.2 plan, and each is marked as such where it appears.
Purely mechanical Rust syntax (for example, whether a helper takes its
argument by reference) may differ in detail; the semantics shown here do not
change.

## Handler signatures at a glance

- Request handler:
  `Fn(Arc<S>, Context, Params, CancellationToken) -> Future<Output = Result<ResultType, LspError>>`
- Notification handler: `Fn(Arc<S>, Context, Params) -> Future<Output = ()>`

`Context` arrives by value and is cheap to clone. `Arc<S>` is your state,
wrapped by the framework â€” you never wrap it yourself.

## Struct state

0.1 stored framework state (the `Documents` store) inside your struct and
handed it back through the required `documents()` getter:

```rust
// 0.1
struct Hello {
    documents: Documents,
}

impl Hello {
    fn new() -> Self {
        Self {
            documents: Documents::new(),
        }
    }
}

impl LanguageServer for Hello {
    fn documents(&self) -> &Documents {
        &self.documents
    }
}
```

In 0.2 the framework owns `Documents`. Your struct carries only your own
application state, and you pass it to `Server::builder`:

```rust
// 0.2
struct State {
    // Only your own application state lives here â€” no Documents,
    // no Context, no framework handles.
}

impl State {
    fn new() -> Self {
        Self {}
    }
}

let server = lspf::Server::builder(State::new())
    // ...registrations...
    .build()?;
```

Every handler receives that state as `Arc<State>`. Handlers reach the
document store through the `Context` parameter: `ctx.documents()` returns a
minimal read-only view (`DocumentsView`, per the 0.2 plan); user code never
constructs `Documents` or `Context` itself.

## Lifecycle handlers

0.1 overrode trait methods, and could shape capabilities by overriding
`server_capabilities()`:

```rust
// 0.1
impl LanguageServer for Hello {
    async fn initialize(
        &self,
        _ctx: &Context,
        _params: InitializeParams,
        _ct: CancellationToken,
    ) -> Result<InitializeResult, LspError> {
        Ok(InitializeResult {
            capabilities: self.server_capabilities(&_params),
            server_info: Some(ServerInfo {
                name: "hello".into(),
                version: None,
            }),
        })
    }

    async fn shutdown(
        &self,
        _ctx: &Context,
        _ct: CancellationToken,
    ) -> Result<(), LspError> {
        Ok(())
    }

    async fn exit(&self, _ctx: &Context) {}
}
```

0.2 moves these to three dedicated builder hooks. `ServerCapabilities` are
generated by the framework from your registrations â€” hooks cannot replace
them, and `server_capabilities()` no longer exists:

```rust
// 0.2
let server = lspf::Server::builder(State::new())
    .on_initialize(on_initialize)
    .on_shutdown(on_shutdown)
    .on_exit(on_exit)
    .build()?;

// Request-handler shape; runs after the Workspace is established and
// before capabilities are returned. Its optional ServerInfo is combined
// with the framework-generated capabilities.
async fn on_initialize(
    _state: Arc<State>,
    _ctx: Context,
    _params: InitializeParams,
    _ct: CancellationToken,
) -> Result<Option<ServerInfo>, LspError> {
    Ok(Some(ServerInfo {
        name: "hello".into(),
        version: None,
    }))
}

// Request-handler shape; runs before the engine enters shutting-down.
async fn on_shutdown(
    _state: Arc<State>,
    _ctx: Context,
    _params: (),
    _ct: CancellationToken,
) -> Result<(), LspError> {
    Ok(())
}

// Notification-handler shape; runs before the engine computes the exit
// outcome. The outcome itself is protocol-owned and cannot be replaced.
async fn on_exit(_state: Arc<State>, _ctx: Context, _params: ()) {}
```

Related moves:

- The 0.1 `initialized` notification override registers as
  `.notification::<lspf::types::notification::Initialized>(handler)`.
- Initialization-dependent route or capability registration does not go in
  `on_initialize`; it goes in the single synchronous
  `.configure_initialize(|params, registrar| Ok(()))` transaction, which
  runs before the Router is frozen and capabilities are computed.

## The didOpen hook (and other document notifications)

0.1 overrode the trait method. The framework updated the document store
around your override:

```rust
// 0.1
impl LanguageServer for Hello {
    async fn text_document_did_open(
        &self,
        ctx: &Context,
        params: DidOpenTextDocumentParams,
    ) {
        // Framework state was already updated; react here.
    }
}
```

0.2 registers a post-mutation hook for the built-in notification through
the ordinary `notification` registration. Registering a built-in
notification name records the hook instead of a Router entry:

```rust
// 0.2
let server = lspf::Server::builder(State::new())
    .notification::<lspf::types::notification::DidOpenTextDocument>(on_did_open)
    .build()?;

async fn on_did_open(
    _state: Arc<State>,
    ctx: Context,
    params: DidOpenTextDocumentParams,
) {
    // Runs after the built-in mutation: ctx.documents() already contains
    // the opened document.
}
```

For every built-in notification the ordering is fixed:

1. decode typed parameters;
2. perform built-in validation and mutation;
3. invoke the user hook.

The hook observes post-mutation state and cannot suppress, roll back, or
reorder the built-in. A decode or validation failure means the hook never
runs. Registering two hooks for the same built-in notification is a
`BuildError`. The other 0.1 document overrides map as follows:

- `text_document_did_change` and `text_document_did_close` are also built-in
  mutations; each becomes one `.notification::<N>(handler)` post-mutation
  hook with the same ordering guarantees as didOpen.
- `text_document_did_save` has no built-in mutation behind it, so
  `.notification::<lspf::types::notification::DidSaveTextDocument>(handler)`
  registers an ordinary Router notification handler rather than a
  post-mutation hook.

## Diagnostics

0.1 pushed diagnostics through a helper on the borrowed `&Context`:

```rust
// 0.1
async fn text_document_did_open(
    &self,
    ctx: &Context,
    params: DidOpenTextDocumentParams,
) {
    ctx.publish_diagnostics(PublishDiagnosticsParams {
        uri: params.text_document.uri,
        version: Some(params.text_document.version),
        diagnostics: vec![/* ... */],
    });
}
```

In 0.2, outgoing traffic goes through the typed `Client` handle exposed by
`Context`. A typed notification is encoded and offered to the engine-owned
outbound queue without allocating a request ID:

```rust
// 0.2
use lspf::types::notification::PublishDiagnostics;

async fn on_did_open(
    _state: Arc<State>,
    ctx: Context,
    params: DidOpenTextDocumentParams,
) {
    let result = ctx
        .client()
        .notify::<PublishDiagnostics>(PublishDiagnosticsParams {
            uri: params.text_document.uri,
            version: Some(params.text_document.version),
            diagnostics: vec![/* ... */],
        })
        .await;
    if let Err(err) = result {
        tracing::warn!(error = %err, "publish diagnostics failed");
    }
}
```

`Client` is also the handle for typed server-to-client requests
(`Client::request`), which reserve an engine-owned ID and await the
correlated response. `Context` may additionally grow convenience helpers
that delegate to `Client`; it remains the one framework-state parameter
every handler receives.

## stdio serve

0.1 constructed the server value inline and served it over stdio:

```rust
// 0.1
#[tokio::main]
async fn main() -> lspf::Result<()> {
    tracing_subscriber::fmt()
        .with_writer(std::io::stderr)
        .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
        .init();
    lspf::stdio(Hello::new()).serve().await
}
```

0.2 keeps the `lspf::stdio(server).serve()` call shape, but the argument is
now a built `Server`:

```rust
// 0.2
#[tokio::main]
async fn main() -> lspf::Result<()> {
    tracing_subscriber::fmt()
        .with_writer(std::io::stderr)
        .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
        .init();

    let server = lspf::Server::builder(State::new())
        .notification::<lspf::types::notification::DidOpenTextDocument>(on_did_open)
        .build()
        .expect("static registrations are valid");
    lspf::stdio(server).serve().await
}
```

The custom-transport escape hatch is likewise retained under the 0.2 plan as
`lspf::serve(server, transport).await?`, taking the built `Server` and any
`Transport` implementation. One `Server` owns exactly one connection; a
second connection requires a second `Server`.

## Concurrency limit

0.1 configured the in-flight handler cap at the serve entry point or on the
stdio builder, with `DEFAULT_CONCURRENCY_LIMIT` (64) as the implicit value:

```rust
// 0.1
lspf::serve(server, transport).await?; // uses DEFAULT_CONCURRENCY_LIMIT (64)
lspf::serve_with_limit(server, transport, 128).await?;
lspf::stdio(server).concurrency_limit(128).serve().await?;
```

In 0.2 the cap is part of the fixed, framework-owned default stack â€”
`PanicIsolation â†’ Tracing â†’ ConcurrencyLimit â†’ user layers â†’ RouterService`
â€” and is configured on the `ServerBuilder`, not on the transport or serve
call. The fixed stack and its semantics are locked by ADR 0019; the tuning
knob itself is the planned 0.2 builder API:

```rust
// 0.2
let server = lspf::Server::builder(State::new())
    .concurrency_limit(128)
    .build()?;
```

Notes on the 0.2 semantics:

- Omitting `concurrency_limit` keeps the framework default cap.
- A limit of `0` fails validation with `BuildError::InvalidConcurrencyLimit`.
- Transport builders no longer carry the knob: business policy moves to the
  `ServerBuilder`.
- The permit is acquired before any user layer runs and is held until the
  call's result completes; tracing spans include the wait for a permit.
- There is no opt-out switch: the 0.1-era `.no_default_layers()` idea is
  gone, and `ConcurrencyLimit` cannot be removed or reordered.

## lspf-hello coverage

The `lspf-hello` example (0.1) uses the following surface; each row points
at its mapping above.

| 0.1 usage in `lspf-hello` | 0.2 mapping |
| --- | --- |
| `impl LanguageServer for Hello` | Removed trait; registrations move to `Server::builder` (all sections) |
| `struct Hello { documents: Documents }` + `Documents::new()` | Framework-owned store; user struct holds only app state (Struct state) |
| `fn documents(&self) -> &Documents` | Removed getter; handlers use `ctx.documents()` (Struct state) |
| `text_document_did_open` override | `.notification::<DidOpenTextDocument>(handler)` post-mutation hook (The didOpen hook) |
| `ctx.publish_diagnostics(params)` | `ctx.client().notify::<PublishDiagnostics>(params)` (Diagnostics) |
| `lspf::stdio(Hello::new()).serve().await` | `lspf::stdio(server).serve().await` with a built `Server` (stdio serve) |
| `lspf::Result` return from `main` | Unchanged; `serve()` still returns `lspf::Result<()>` (stdio serve) |
| `lspf::types::{...}` imports | Unchanged; `lspf::types` still re-exports `lsp-types` |
| `#[tokio::main]` + `tracing_subscriber` init | Unchanged; not lspf API |

## Full 0.2 lspf-hello

For reference, the complete 0.2 shape of the example:

```rust
use std::sync::Arc;

use lspf::types::notification::{DidOpenTextDocument, PublishDiagnostics};
use lspf::types::{
    Diagnostic, DiagnosticSeverity, DidOpenTextDocumentParams, Position,
    PublishDiagnosticsParams, Range,
};
use lspf::Context;

struct State;

impl State {
    fn new() -> Self {
        Self
    }
}

async fn on_did_open(_state: Arc<State>, ctx: Context, params: DidOpenTextDocumentParams) {
    let result = ctx
        .client()
        .notify::<PublishDiagnostics>(PublishDiagnosticsParams {
            uri: params.text_document.uri,
            version: Some(params.text_document.version),
            diagnostics: vec![Diagnostic {
                range: Range {
                    start: Position {
                        line: 0,
                        character: 0,
                    },
                    end: Position {
                        line: 0,
                        character: 0,
                    },
                },
                severity: Some(DiagnosticSeverity::INFORMATION),
                source: Some("lspf-hello".into()),
                message: "lspf saw this document open".into(),
                ..Diagnostic::default()
            }],
        })
        .await;
    if let Err(err) = result {
        tracing::warn!(error = %err, "publish diagnostics failed");
    }
}

#[tokio::main]
async fn main() -> lspf::Result<()> {
    tracing_subscriber::fmt()
        .with_writer(std::io::stderr)
        .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
        .init();

    let server = lspf::Server::builder(State::new())
        .notification::<DidOpenTextDocument>(on_did_open)
        .build()
        .expect("static registrations are valid");
    lspf::stdio(server).serve().await
}
```
